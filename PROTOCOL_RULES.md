# Protocol Rules

## Status: Enforced

These rules are not suggestions. Violating them will cause integration checks to fail and block merges.

---

## 1. Runtime Authority

- **Rust is the authoritative runtime.**
- Python, C++, and JavaScript are wrappers or clients ONLY.
- Any logic that touches consensus, cryptography, storage, or receipts must be implemented in Rust.

## 2. Canonical Types

- **All receipts use `CanonicalReceipt` from `rust/src/protocol/receipt.rs`.**
- No agent may define its own receipt type.
- No agent may add fields to the receipt without updating the shared schema.
- Receipt serialization is canonical JSON with SHA-256 payload hashes.

## 3. Deterministic Hashes

- **All hashes are SHA-256.**
- Hash computation must be deterministic (same input → same output, every time).
- No non-deterministic serialization (e.g., unordered HashMap JSON) may be hashed directly.

## 4. API Versioning

- **Every HTTP response must include `X-API-Version`.**
- The version value is `SemVer::CURRENT` from `rust/src/protocol/types.rs`.
- Breaking changes increment the major version.
- All API routes must be version-prefixed or version-header-aware.

## 5. No Mock Cryptography

- **No placeholder signatures, no dummy keys, no `#[cfg(test)]` bypasses in production code.**
- Tests may use deterministic test vectors, but production paths must use real ed25519 / SHA-256.

## 6. No Placeholder Responses

- **Every endpoint must return real data or a real error.**
- `todo!()`, `unimplemented!()`, and hardcoded demo values are forbidden in production code.

## 7. No Duplicate Networking Layers

- **One networking abstraction: `libp2p` behind the `network` feature flag.**
- Agent 2 (Solana Query) uses HTTP for Solana RPC, but must not spin up its own P2P layer.
- Agent 3 (Proof) uses HTTP for IPFS / Solana anchoring, but must not spin up its own P2P layer.

## 8. No Duplicate Storage Abstractions

- **One storage abstraction: `rocksdb` via `rust/src/storage.rs`.**
- No agent may open its own RocksDB instance or use ad-hoc file writes for state.
- Receipts, proofs, and query caches must all go through the shared storage interface.

## 9. No Silent Fallback Behavior

- **Explicit defaults only.**
- The `CanonicalConfig` system uses `config::Config` with visible defaults.
- If a required environment variable is missing, the system must fail loudly at startup.
- No hidden autogenerated state (no auto-created directories, no implicit file writes).

## 10. No Hidden Autogenerated State

- **All state mutations must be explicit, logged, and receipted.**
- If the system writes to disk, it must produce a receipt.
- If the system changes config, it must be visible in `CanonicalConfig`.

## Agent Boundaries

### Agent 1 — Core Runtime Stabilization
**Territory:** `crypto.rs`, `storage.rs`, `consensus.rs`, `network.rs`, `metrics.rs`, `security.rs`, `health.rs`
**Forbidden:** UI, product concepts, Solana-specific logic, proof-specific logic

### Agent 2 — Solana + Query + API Layer
**Territory:** `solana_agent/` module
**Forbidden:** P2P networking, consensus algorithm changes, storage abstraction changes, proof system changes

### Agent 3 — Proof + Provenance + MemGas
**Territory:** `zkproof.rs`, `protocol/receipt.rs` (provenance extensions), `protocol/config.rs` (proof config)
**Forbidden:** UI, consensus algorithm changes, storage abstraction changes, query engine changes

---

## Integration Discipline

1. Run `./integration-check.sh` before every merge.
2. If any check fails, stop feature work and fix stability.
3. All three agents must pass the same `cargo test --test daily_integration`.
4. No agent may merge code that breaks another agent's compilation.

## Receipt Format (v0.1.0)

```rust
pub struct CanonicalReceipt {
    pub version: SemVer,
    pub timestamp: u64,
    pub source: AgentId,
    pub receipt_type: ReceiptType,
    pub payload_hash: [u8; 32],
    pub merkle_root: Option<[u8; 32]>,
    pub signature: Option<Vec<u8>>,
    pub description: String,
    pub payload: serde_json::Value,
}
```

**Types:**
- `Execution` — Agent 1 (block production, state transition)
- `QueryResult` — Agent 2 (Solana RPC query results)
- `Proof` — Agent 3 (zk proof generation)
- `Anchor` — Agent 3 (Solana on-chain anchor)
- `Audit` — Any agent (security / compliance event)

---

## Config Hierarchy

1. `LLM_MINA_` prefixed environment variables (highest priority)
2. `config.toml` file (if present)
3. Explicit defaults in `CanonicalConfig::from_env_or_default()` (lowest priority)

No hidden defaults. Every default value is visible in `rust/src/protocol/config.rs`.
